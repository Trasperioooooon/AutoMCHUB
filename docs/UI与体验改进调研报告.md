## §1 浅色主题过亮 → 「暖纸」色板

### 现状与病因

当前浅色 token（`app.css:41-67`）：

```css
--bg: #f1f4ef;        /* 页面底：冷调浅绿灰 */
--surface: #ffffff;   /* 卡片：纯白 */
--surface-2: #f6f9f4; --sunken: #e7ece5;
--text: #1b201c;      --muted: #5a675e;
```

「过亮」的成因有两个，都不是错觉：

1. **纯白 `#ffffff` 卡片占了屏幕大部分面积**。上一轮为了摆脱「灰压灰」选择了最大反差的白卡方案，矫枉过正——纯白的相对亮度是 1.0（顶格），大面积使用在普通室内光下就是眩光源。成熟的浅色系统（macOS、Notion、claude.ai）卡片层几乎都停在 `#fafafa~#fcfcf9` 一带，刻意不触顶。
2. **底色偏冷**（`#f1f4ef` 带蓝绿相），冷调会在主观上进一步放大「刺眼/苍白」感；暖相（偏黄红）的同亮度色会显得柔和。

### 方案（网络调研 + 对比度实测）

调研了三类锚点（详值见附录 A）：Radix Colors 的 Sand/Sage/Olive 暖中性十二级阶梯（官方仓库原值）、claude.ai 的象牙纸底 `#faf9f5`、Notion 的暖米侧栏 `#f7f6f3`。一个反直觉的发现：Radix 的「暖中性」其实非常克制（sand3 的 RGB 通道差仅 2），**真正有「纸感」的产品底色都比它更暖**——所以直接借 Sand 不够，要落在 Sand 与 claude.ai 象牙纸之间。

给出两套候选（完整 token 与全部对比度实测见附录 A），**推荐方案 B「暖沙中性」**：

| Token | 现值（冷绿） | 方案 B 暖沙中性 |
|---|---|---|
| `--bg` | `#f1f4ef` | `#efece2`（暖米纸底） |
| `--surface` | `#ffffff` 纯白 | `#fbfaf5`（暖白，弃纯白，亮度降 ~4.5%） |
| `--surface-2` | `#f6f9f4` | `#f5f3ea` |
| `--sunken` | `#e7ece5` | `#e6e2d6` |
| `--border` / strong | `rgba(22,34,25,.12/.24)` | `rgba(62,55,32,.13/.26)`（底变暗，alpha 微升保发丝可见） |
| `--text` | `#1b201c` | `#221f18`（暖黑） |
| `--muted` | `#5a675e` | `#625d4e` |
| `--accent-text` / `--err-text` / `--accent-solid` | — | 全部不变（绿、红在暖底上实测达标） |

推荐 B 而非 A（暖米微绿）的理由：无彩暖沙让「纸就是纸」，草方块绿只属于强调色——绿字绿按钮不被绿调底色稀释，品牌色反而更清；且 `#fbfaf5` 与 claude.ai 的 `#faf9f5` 几乎同族，是经过大规模产品验证的安全区。

关键对比度实测（WCAG 相对亮度公式，向下取整）：text/surface **15.7:1**，muted/surface **6.2:1**（比现版纯白底的 5.9 反而更高——底暗了、可读性不降反升），accent-text/surface 4.9:1，err-text 7.3:1，按钮白字 4.0:1。**全部一次达标。**

**深色主题结论：不需要联动改。** 深色三层绿倾向本就接近感知阈值，切换连续性由强调绿承接；Notion（暖浅 + 冷灰深）就是同构先例。想极致统一时可把深色各拉暖半档（`#131611→#1c1f19→#252921`），属可选打磨。

### 实施注意

- 只动 `:root[data-theme="light"]` 块的十几个 token，组件零改动（v3 已全 token 化）。
- 终端容器 `--term-bg` 双主题恒深，不受影响。
- `renderMotd()` 浅色压暗系数（×0.55）与基色 `#565e57` 需要随新 muted 微调复验。
- 深色主题是否联动调暖，以调研结论为准（初步判断：深色现为绿调中性，与暖纸切换观感割裂风险低，可不动）。

## §2 服务器创建位置：默认在哪、能不能选

### 现状（代码事实）

- **默认位置写死**：`internal/inst/install.go` `validateCreate()` 中 `dir := filepath.Join(app.ServersDir, req.Name)`，而 `app.ServersDir = {AutoMCHUB.exe 所在目录}\servers`（`app.go:54`）。也就是说：**你的服务器全部在 `桌面\AutoMCHUB\servers\{实例名}\`**。
- **实例名 = 文件夹名**：目录直接用实例名命名（允许中文），这正是当初「Mohist/Banner 中文实例名必崩」的根因——现在的兜底是创建时拒绝非 ASCII 名（`install.go:54`），治标不治本。
- **路径透明度差**：整个创建向导从头到尾不显示目标路径；创建完成后，路径只在两处露出——实例详情「内存/启动」标签页的小字提示、全局设置副标题的「数据目录」。好在「📁 打开目录」按钮在卡片和详情页都有。
- **实例发现机制**：`inst.go` `NewManager()` 启动时**只扫描 `servers\` 这一个目录**下含 `instance.json` 的子目录。这是「自选位置」的结构性约束：换目录不只是改一个 Join，还牵动加载、删除安全校验（`Delete()` 用 `filepath.Rel(ServersDir, dir)` 防越界，自定义路径会被它拒绝）、备份路径等。

### 为什么值得做（不只是偏好问题）

1. **磁盘空间**：整合包服 + 世界存档动辄 5~20 GB，桌面通常在 C 盘；用户理应能放到 D/E 盘。
2. **Program Files 陷阱**：数据自包含于 exe 目录的设计，一旦用户把 exe 放进 `Program Files` 就会遭遇写权限问题。数据目录可配置是逃生通道。
3. **根治 Mohist 中文名**：目录名与显示名解耦后（目录用 ASCII slug），中文实例名对 Mohist/Banner 也能用。

### 分层方案（建议按序落地）

**T1 · 路径透明化（半小时，无风险）**
向导第 4 步「基本设置」显示目标路径（跟随实例名实时变化）：`将创建于 D:\...\servers\我的服务器`；创建完成页与详情页头部补路径 + 打开按钮。回答了「默认创建在什么地方」这个最基本的疑问。

**T2 · 全局「服务器存放目录」可改（半天）**
全局设置新增「存储」项：`serversDir` 写入 config.json，空 = 默认。变更只对**新建**实例生效；Manager 改为扫描「默认目录 + 配置目录 + 历史目录列表（config 记录曾用根目录）」，旧实例原地继续工作，避免搬家大 IO 的风险。备份目录 `backupsDir` 同样处理。

**T3 · 每实例自定义路径 + 名目录解耦（1~1.5 天）**
- 向导第 4 步加「高级：自定义存放位置」折叠项：默认收起显示 T1 的路径，展开后是路径输入框 + 「浏览…」按钮。
- **目录名 slug 化**：默认目录名 = 实例名的 ASCII 安全化（中文名→拼音或 `server-{时间戳}`），`instance.json` 里的 `name` 保持中文显示名。Manager 的 map 键继续用 name，Dir 从扫描处得出（现有结构已支持 Dir ≠ name）。Mohist/Banner 的限制从「拒绝中文名」降级为「检查最终路径含非 ASCII 才拒绝」（exe 放中文路径下仍会命中，提示搬家）。
- **删除安全语义**：`Delete()` 的 Rel 校验改为「目录内必须存在本实例的 instance.json 才允许 RemoveAll」，对任意合法位置都安全。

**「浏览…」文件夹选择器的技术选型**（WebView2 的 JS 拿不到本地路径，必须走后端）：

| 方案 | 优缺点 |
|------|--------|
| ✅ 推荐：Win32 `IFileOpenDialog`（`FOS_PICKFOLDERS`）COM 调用 | 现代系统对话框；纯 Go 可做（社区有 MIT 的 go-common-file-dialog 可参考实现，~150 行 COM 封装，无 CGO） |
| 兜底：`powershell -STA` + `FolderBrowserDialog` | 零依赖但首启慢 ~1s、窗口可能被遮挡 |
| ❌ `<input webkitdirectory>` | 浏览器沙箱拿不到真实路径，不可行 |

新端点 `POST /api/pickdir` 弹系统对话框并返回所选路径。**必须限制仅 localhost token 会话可调用**——手机经局域网管理时点「浏览」会在主机上弹窗，既无意义又易造成困惑。

## §3 server.properties：从 17 项到全量中文目录

### 现状（代码事实）

- 「常用设置」标签页由 `app.js` 的 `COMMON_PROPS` 白名单驱动，目前 **17 项**：online-mode、allow-flight、pvp、difficulty、gamemode、max-players、server-port、motd、view-distance、simulation-distance、spawn-protection、white-list、enable-command-block、spawn-monsters、hardcore、force-gamemode、level-seed。（你要的「难度」其实已在其中——但创建向导只露出正版验证/允许飞行两个开关，第一印象就是「只能改这两个」。）
- 「全部配置」标签页是裸键值表：英文键名、无任何说明，等于把用户赶去搜索引擎——恰是你想消灭的麻烦。
- 官方 server.properties 全量约 **60 项**，覆盖率不到 1/3；且现有 17 项无分组、无搜索、数字项无范围约束、不标注「需重启」。

### 设计方案

1. **数据驱动的全量目录**：把 COMMON_PROPS 扩展为约 60 项的 `PROPS_CATALOG`（独立常量，含：键名、控件类型、枚举值+中文名、默认值、中文名称、一句话中文说明、版本适用区间、是否建议重启）。
2. **分组呈现**：基础与玩法 / 世界生成 / 玩家与权限 / 性能 / 网络与远程 / 资源包 / 安全与杂项——组内锚点导航（页首一排分组 chips）+ 顶部搜索框（按中文名/键名过滤）。
3. **版本感知**：目录标注了适用版本的项，按实例 MC 版本隐藏不适用项（如 1.12.2 不显示 simulation-distance），避免写入无效键。
4. **「全部配置」页升级**：同一份目录用来给 raw 表格的已知键补一列中文名——目录之外的陌生键（Paper/Purpur 自有配置等）保持原样。
5. **创建向导克制地加两项**：难度、默认游戏模式（下拉，一行放下），并加一句「更多几十项设置可在创建后的『常用设置』中修改」——解决第一印象问题，又不把向导变成长表单。

### 全量目录与易踩坑项

已按官方中英文维基整理出**约 60 项 × 7 组**的全量中文目录（键名/类型与可选值/默认值/中文名/一句话说明/版本备注），全表见附录 B，可直接作为 `PROPS_CATALOG` 的实现数据。整理过程中有一个**改变方案优先级的发现**：

> **1.21.9 起 `pvp`、`spawn-monsters`、`allow-nether`、`enable-command-block` 四项被移除**，迁移为游戏规则（gamerule）。这意味着现有 17 项白名单里就有 **3 项在 26.x 实例上是死键**——写了服务端也不认。所以上文的「版本感知」不是加分项，是**正确性需求**。对这类已迁移项，面板应在新版本实例上把控件类型切换为 gamerule 模式：运行中通过控制台发 `/gamerule` 即时生效，停机时置灰并提示「启动后可改」。

**易踩坑项清单**（逐条核实过版本号，全部应编入目录的说明文案，替用户省掉搜索）：

1. **enforce-secure-profile（1.19+）与离线模式**：原版离线服不会因它踢人，但代理/插件环境保留 true 会出现「无法验证安全档案」类踢出。**建议创建流水线联动：onlineMode=false 时显式写入 enforce-secure-profile=false**（对 `install.go` 步骤 4 是一行改动）；面板检测到离线模式时也自动联动此项。
2. **spawn-animals / spawn-npcs 已于 1.21.2 移除**（无一对一替代）；注意 spawn-monsters 是 1.21.9 才移除的，两批时间不同，需分别按版本隐藏。
3. **level-type 写法随版本变化**：1.19+ 用命名空间 ID（`minecraft:normal/flat/large_biomes/amplified`…）；1.16–1.18.2 写 `default/flat/largeBiomes/amplified`；1.12.2 用大写 `DEFAULT/FLAT/…`。目录需按版本给不同枚举集。
4. **simulation-distance 是 1.18 才有的**：1.17 及以前不存在（现有白名单已包含此项，在老版本实例上同属死键）。
5. 较新键引入版本：accepts-transfers（1.20.5）、region-file-compression（1.20.5）、pause-when-empty-seconds（1.21.2）、bug-report-link（1.21）、hide-online-players（1.18）、initial-enabled/disabled-packs（1.19.3）、chat/command-spam-threshold-seconds（26.2）。
6. 已移除键：max-build-height（1.17）、snooper-enabled（1.18）、previews-chat（仅 1.19–1.19.2 存在）。特别注意：1.21.9 的 enable-command-block → 游戏规则 enableCommandBlocks 后**默认值从 false 翻转为 true**。
7. **white-list 与 enforce-whitelist**：前者决定是否启用白名单（拦新连接，OP 豁免）；后者让重载名单时把已在线的非名单玩家也踢出。**预警：26.3 快照起 white-list 默认值改为 true**——新版实例「进不来服」的工单预定项，面板应在 26.3+ 创建流程里明示白名单状态。

## §4 设置页信息架构：从平铺到分区

### 现状（代码事实）

`renderSettings()`（`app.js:1068`）输出一根**单列长瀑布**：下载源 → CurseForge Key → Java 管理 → 远程访问 → Webhook → 自动更新 → 一行落款 → 退出按钮。全部靠 `<h3>` 分隔，无卡片边界、无图标、控件形态五花八门（radio 卡片/输入行/按钮混排），越往下越像「配置文件的 GUI 化罗列」——与你的体感一致。

### 方案：页内二级导航（HMCL 同构）

保持底部物品栏 4 格不动（它是产品签名），在**设置页内部**引入左侧二级导航：

```
┌──────────────────────────────────────────────────┐
│  OPTIONS                                          │
│  全局设置                                          │
│ ┌─────────┐ ┌───────────────────────────────────┐ │
│ │▶ 下载与网络│ │  下载源                            │ │
│ │  存储位置  │ │  ○ 自动（推荐）  ○ 仅镜像  ○ 仅官方  │ │
│ │  Java     │ │  ───────────────────────────      │ │
│ │  远程访问  │ │  CurseForge API Key               │ │
│ │  通知与集成│ │  [粘贴 Key…            ] [保存]    │ │
│ │  更新与关于│ │                                   │ │
│ └─────────┘ └───────────────────────────────────┘ │
└──────────────────────────────────────────────────┘
```

分组规划（6 组，右栏均为卡片内的「行式控件」）：

| 组 | 内容 |
|----|------|
| 下载与网络 | 下载源三选一、CurseForge API Key |
| 存储位置 | 数据目录展示、服务器存放目录（§2 T2）、备份目录、各目录「打开」按钮 |
| Java | 便携运行时列表、扫描结果、手动添加 |
| 远程访问 | 局域网开关、密码、访问地址（修正端口，见 §6-B1）、防火墙命令、（后续）二维码 |
| 通知与集成 | Webhook URL、事件说明 |
| 更新与关于 | 版本号、检查更新、更新仓库、开源许可、OpenFrp 署名（从页脚小字升为正式条目）、退出程序（红色危险区独立分隔） |

**行式控件统一模式**（借鉴 HMCL/Android 偏好页）：`[图标] 标题 + 副标题说明 ……右侧对齐控件（开关/下拉/按钮）`，全部行等高、说明文案固定在标题下方——消灭「罗列感」的关键不是分页，而是**每一行长得一样**。

顺带收益：Java 列表、防火墙提示这些「长内容」被收进各自分区后，任何一屏的信息密度都可控。

### 5.1 它的骨架长什么样（源码核实）

- **窗口**：无系统边框自绘标题栏（帮助/最小化/关闭三键，无最大化）；主窗口 = 左侧边栏（账户卡片、当前实例卡片、实例列表、下载、设置、多人联机）+ 中央内容；主页极简——整页壁纸 + 右下角大启动按钮 + 切换实例上拉箭头。
- **全局设置 = 页内二级布局**（左子导航 200px + 右内容），分类原文：`全局游戏设置 / Java 管理 /（启动器）通用、外观、下载 /（帮助）帮助、反馈、关于`——与本报告 §4 提议的结构同构，证明该模式在中文 MC 工具用户群里已被大规模验证。
- **控件模式三件套**（所有设置页统一）：①行式控件「标题 + 副标题（解释文案/风险提示/『重启后生效』）+ 右侧开关/下拉」；②**可展开行收起时显示当前值摘要**（内存行收起显示「游戏分配 6.0 GiB」）；③选项行内嵌编辑器（选中「手动选择内存」时行内直接出现滑条 + 输入框）。
- **内存设置的标杆做法**：滑条（220px）+ 精确数字输入框 + 单位标签，下方常驻**实时内存状态条**「已使用 X GiB / 总内存 Y GiB ｜ 游戏分配 Z GiB」，超配即警示「设备仅 X GiB 可用」。
- **实例管理页**：左子导航（游戏设置/自动安装/模组管理/资源包管理/世界管理/原理图管理）+ 左下角固定工具栏，其中「浏览」弹出**文件夹直达菜单**（运行目录/模组/资源包/世界/截图/日志/崩溃报告等 10 个子目录一键直开）、「管理」弹出（重命名/复制实例/导出整合包/清理游戏文件夹…）。
- **版本列表**：图标（正式版=草方块/快照=命令方块）+ 双行（版本号 + 发布时间）+ 类型标签 + 兼容性警告标签；**版本类型过滤器默认「正式版」**；搜索支持 `regex:` 前缀。
- **下载源三档人话**：「自动选择 / 尽量使用官方源 / 尽量使用镜像源（推荐中国大陆用户使用）」——与我们现有三档几乎逐字同构，这条我们已经做对了。
- **主题系统**：浅色/深色/**跟随系统**三态；主题色自定义（3.16 起可从壁纸取色，Material You 式）；背景图**约定优于配置**（程序旁放 `background.png` 或 `bg/` 文件夹即随机壁纸）；字体可换、动画总开关。
- **零状态即主行动**：一个实例都没有时，启动按钮直接变「开始游戏」= 自动装最新正式版并启动，空状态不是死路而是主任务捷径。
